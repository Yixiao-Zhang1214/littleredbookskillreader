### 1. 链接类型校验与内容隔离 (强制拦截)
- **检查链接形式**：确认用户提供的是小红书链接。
- **拦截视频**：你必须先尝试读取或判断该链接是否为视频。如果 URL 中明显带有 `video`，或者你在获取内容时发现它是视频类型，**必须立即停止分析**，并回复用户：“❌ 抱歉，本工具仅支持处理小红书图文笔记，不支持视频内容。”
- **严格内容隔离机制**：当读取到用户新提供的小红书链接时，必须严格执行内容隔离。**仅引入与该特定帖子直接相关的信息。**严禁在处理过程中主动或被动引入任何与当前帖子主题、内容、作者或上下文无关的外部信息、历史对话内容或其他无关数据。所有内容生成、分析和回复必须**完全基于当前小红书链接所指向的帖子本身**，确保信息处理的精准性和上下文独立性。

### 2. 小红书内容获取 (渐进式策略)
- **步骤 2.1 匿名直连尝试**：首先尝试使用基础工具（如 WebFetch 工具或 Python 请求）直接读取该小红书链接的 HTML 源码。
- **步骤 2.2 拦截判定与降级**：分析提取到的内容。如果内容为空、或者明显是“登录/扫码”的重定向页面（触发了反爬），说明直连失败，此时**必须降级使用 Agent Reach**。如果直连成功读取到了笔记的图片或文字，则直接进入下一步（步骤 2.4），无需调用 Agent Reach。
- **步骤 2.3 调用 Agent Reach (如需)**：当直连被拦截时，调用本地已安装的 `agent-reach` 环境（通过 `xhs read` 或相应的 Python SDK）进行带 Cookie 的读取。
  - *环境依赖*：作为本 Skill 的一部分，Agent Reach 已被封装在安装依赖中。
  - *未登录处理*：如果在调用 Agent Reach 时发现未配置 Cookie 或提示未授权，你需要中断流程并向用户输出：*“🚨 小红书触发了强制登录拦截。请在浏览器安装 Cookie-Editor 插件，将小红书网页版的 Header String 复制发给我，我来帮您配置环境。”*
- **步骤 2.4 逐图精读与提取 (严禁下载，严禁幻觉)**：
  - 通过 `window.__INITIAL_STATE__` 或解析返回的 JSON，提取出笔记中包含的**所有图片 URL 列表**。
  - **直接读取能力**：你必须利用自身强大的多模态视觉理解能力（或 OCR 依赖），直接通过网络读取这些图片的 URL，从中提取文本信息。**绝对不允许将图片下载保存到本地临时目录。**
  - **严禁无中生有 (Anti-Hallucination)**：你必须**忠实于原文和原图**。只能提取图片或文本中真实存在的 Skill 名称和对应的 GitHub/安装链接。如果在笔记中完全没有找到任何具体的 Skill 名称或链接，你**绝对不允许**凭空捏造、编造或根据“产品经理常用工具”自行猜测 Skill！如果没有找到，必须直接向用户回复：“未在这篇笔记中提取到明确的 Skill 安装信息。”

### 4. 系统性风险评估 (结合源码与业务)
对于提取出的每一个技能，结合上述**源码审查结果**与业务逻辑，严格依据以下 **6 个维度** 进行风险分析，并为每个维度评定等级（高/中/低）：
1. **合规性 (Compliance)**：是否符合国家法律法规（如《个人信息保护法》）、行业规范及平台使用条款。
2. **安全性与权限范围 (Security & Permission Scope)**：
   - 评估它需要读取/写入哪些文件？运行哪些命令？是否需要网络访问？权限是否做到了最小化？
   - 是否触发了上述的**红旗清单**？是否存在代码投毒或非授权越权操作。
3. **伦理影响 (Ethics)**：是否会产生算法偏见、诱导不良价值观、或替代人类进行具有道德争议的决策。
4. **技术成熟度与生态 (Technology & Ecosystem)**：
   - 实现逻辑是否稳定，是否过度依赖易失效的第三方 API。
   - **执行 100/3 法则校验**：尝试通过 GitHub API 或搜索确认该仓库是否满足“100次以上下载/Star，且有3个月以上发布历史”。如果不满足，提示“缺乏社区验证（小白鼠风险）”。
   - 是否被判定为“纯套壳 Skill”。
   - **信任层级评估**：判断作者是否知名，来源是否为官方仓库。
5. **用户体验风险 (UX)**：交互设计是否容易导致用户误解（如“幻觉”输出），使用门槛是否过高。
6. **业务适配性 (Business Fit)**：功能定位是否与目标受众（如产品经理）的职责和业务目标相冲突。

基于上述维度的综合表现，为该技能得出最终的 **综合风险评级**：
- **🟢 LOW (低风险)**：例如仅记笔记、格式化等。常规审核通过即可安装。
- **🟡 MEDIUM (中风险)**：涉及文件操作、浏览器或 API 调用。需完整的代码审查。
- **🔴 HIGH (高风险)**：涉及凭证读取、交易、系统命令。**必须由人类用户审批**。
- **⛔ EXTREME (极度危险)**：触碰红旗清单、修改安全配置或请求 root 权限。**绝对禁止安装**。

并为每一项提出相应的 **风险缓解建议**。

### 5. 跨平台适配与运行兼容性评估机制
在得出最终结论前，你必须针对当前使用的 Trae 平台（以及可能适配的 WorkBuddy 等平台），实施以下全面的兼容性评估。受限于安全沙箱原则（禁止动态执行未受信任代码），此评估基于**深度静态代码分析与逻辑推演**进行：

1. **平台特性分析 (Platform Characteristics)**：
   - *评估指标*：检查源码是否依赖特定 IDE 的私有 API（如 Cursor、VSCode 专属插件 API），或者是否使用了通用的 MCP (Model Context Protocol) 协议。
   - *判定标准*：如果纯基于 Prompt 或通用 MCP 协议，则完美适配 Trae/WorkBuddy；如果硬编码了不兼容的宿主 API，则判定为「不适配」。
2. **依赖项检查 (Dependencies Check)**：
   - *评估指标*：审查代码库中的 `package.json`、`requirements.txt` 或环境配置说明。
   - *判定标准*：评估依赖的复杂度与安装成本。无外部依赖或仅依赖内置环境为优；如果需要安装大量系统级底层依赖（如 C++ 编译工具链），则标记为「适配成本过高」。
3. **接口兼容性验证 (Interface Compatibility)**：
   - *评估指标*：分析 Skill 的输入输出规范及触发条件。
   - *判定标准*：检查其 I/O 是否符合 Trae/WorkBuddy 的标准文本交互流。如果强制要求特殊的 Webhook 回调、复杂的本地特定端口通信、或图形化弹窗，而当前环境未明确保障支持，则判定为「接口存在兼容风险」。
4. **功能完整性推演 (Functional Completeness)**：
   - *评估指标*：对比“小红书笔记宣称功能”与“实际源码业务逻辑路径”。
   - *判定标准*：通过静态代码阅读，确认核心功能链条是否闭环。如果发现只有一套外壳 Prompt 而无实际工具支撑，或者核心功能重度依赖不可用的外部第三方私有服务，判定为「功能不完整/纯套壳」。
5. **性能基准与资源消耗预估 (Performance Benchmark)**：
   - *评估指标*：代码中的循环逻辑、文件读取范围、并发请求设置。
   - *判定标准*：如果发现无限制的 `os.walk` 遍历整个磁盘、海量并发发包、或缺乏超时控制的复杂正则匹配，判定为「存在严重性能瓶颈或资源枯竭风险」。
